## Q1 — Did the recency cap bound the archive-serving feeds? ✅ Yes

The cap is applied uniformly in the orchestrator and bounded every oversized feed. With a 7-day
window:

| Source | Fetched | Kept | Note |
|---|---:|---:|---|
| **OpenAI Blog** | **1012** | **7** | the worst archive-dumper this run — fully bounded |
| **Eugene Yan** | **210** | **0** | archive blog, nothing inside 7 days |
| Google DeepMind Blog | 100 | 2 | |
| The Register | 50 | 12 | hit the item ceiling |
| Cursor Changelog | 50 | 2 | |
| Ars Technica AI | 20 | 12 | hit the item ceiling |
| Hacker News (front page) | 20 | 12 | hit the item ceiling |
| Simon Willison's Weblog | 30 | 12 | hit the item ceiling |

Both halves of the cap proved out: the **age window** zeroed pure-archive feeds (Eugene Yan
210→0, VentureBeat 7→0) and trimmed OpenAI Blog 1012→7; the **item ceiling** (12) capped the
high-volume live feeds (The Register, Ars Technica, HN, Simon Willison). One bad/oversized feed
can no longer dominate or blow up a run. Failures stay isolated per source (no source error this
run).

## Cost & caveats

- **Cost:** ~$15 of `claude-opus-4-8` enrichment for 120 events (one full enrichment + entity/
  relationship extraction per de-duplicated event). Roughly linear in `enriched_events`.
- **Recency window:** run used **7 days**; `.env.example` ships a more permissive **30-day**
  default. 7 days is aggressive — it zeroed slower feeds (Eugene Yan, the release feeds) entirely.
  For routine use 14–30 days gives the convergence window more overlap to work with; 7 days is a
  good "what changed *this week*" setting.
- **Advisories window:** `github_advisories` is bounded by the same recency cap, so the fastapi
  advisory (2 found, both > 7 days) was filtered out. For an Early-Warning source a longer
  dedicated window would be worth considering later (out of scope for M6.5 — noted, not changed).
- **Throwaway DB:** ran against `/tmp/m6.5.db` with Neo4j + Qdrant rebuilt from zero; no
  production state was touched.
